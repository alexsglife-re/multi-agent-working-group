## Why

The Owner has authorized `v1.0.0 — Progressive Leader State Profiles` to
replace `v0.4.18` as the current public version through an exact reviewed tag
and formal GitHub Release. The repository must first convert every public
status surface and deterministic validation rule without weakening the
documented pilot limitations or implying deployment.

## What Changes

- Convert README, changelog, roadmap, TODO, validation, traceability,
  installation, adapter, template, and deterministic version markers from the
  v1.0.0 development state to the v1.0.0 public state.
- Preserve `v0.4.18` as historical release evidence and declare no next
  development target unless the Owner separately authorizes one.
- Keep public disclosures that structural thresholds remain provisional
  classification baselines, size bands remain warning-only, and Hierarchical
  root/card/shard storage is not implemented.
- Define the documented release date as the actual `America/Los_Angeles`
  calendar date when the annotated tag is created, while recording the GitHub
  Release `publishedAt` timestamp independently.
- Complete fresh C1-C4 review, validation, commit, push, status, and archive
  gates, then freeze the reviewed C4 archive commit as the tag target.
- Create and verify annotated tag `v1.0.0`, then publish one formal GitHub
  Release titled exactly `v1.0.0` with the reviewed changelog section as its
  exact body, `draft=false`, and `prerelease=false`.
- Exclude deployment, package publication, social/community publication,
  force-push, history rewrite, protected-branch bypass, secret/auth/permission
  changes, schema changes, destructive actions, and unrelated GitHub metadata.

## Capabilities

### New Capabilities

None.

### Modified Capabilities

- `public-release-status`: Make v1.0.0 the current public version, preserve its
  explicit pilot and Hierarchical limitations, and define exact tag, Release,
  publication, metadata-verification, and no-deployment closeout behavior.

## Impact

The change affects release-facing Markdown, the three v1.0.0 profile template
version labels, `scripts/validate-local.sh`, the accepted
`public-release-status` specification after sync, OpenSpec archive evidence,
the `main` branch, annotated tag `v1.0.0`, and the repository's formal GitHub
Release list. It changes no runtime API, dependency, repository setting,
credential, schema, package registry, or deployment target.

The tracked implementation allowlist is exactly: `README.md`, `CHANGELOG.md`,
`docs/ROADMAP.md`, `docs/TODO.md`, `docs/VALIDATION.md`,
`docs/INSTALLATION.md`, `docs/ADAPTERS.md`, `references/TRACEABILITY.md`,
`templates/compact-leader-state.md`, `templates/standard-leader-state.md`,
`templates/successor-opportunity-skeleton.md`, `scripts/validate-local.sh`,
the active `openspec/changes/publish-v1-0-0-release-status/**` artifacts, the
sync target `openspec/specs/public-release-status/spec.md`, and the eventual
`openspec/changes/archive/<action-date>-publish-v1-0-0-release-status/**`
archive. Every other tracked path, including
`templates/review-packet-cleanup-checklist.md`, is out of scope unless a new
reviewed scope decision explicitly adds it.
